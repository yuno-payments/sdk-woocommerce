# Domain Vocabulary

## Yuno Platform

| Term | Definition |
|---|---|
| **Yuno** | Payment orchestration platform that aggregates multiple payment providers and methods under a single API |
| **Yuno Web SDK** | Client-side JavaScript SDK (v1.6) that renders payment forms, handles tokenization, 3DS challenges, and APM flows |
| **Yuno API** | Server-side REST API at `api[-env].y.uno` for customer, session, and payment management |
| **Full SDK / SDK_CHECKOUT** | Yuno workflow type where the SDK manages the complete checkout experience (method selection, form rendering, tokenization) |
| **Account ID** | Merchant identifier in the Yuno platform |
| **Public API Key** | Frontend-safe key used to initialize the Yuno SDK; its prefix determines the API environment |
| **Private Secret Key** | Server-only key for authenticating with the Yuno API; never exposed to the frontend |

## Payment Lifecycle

| Term | Definition |
|---|---|
| **Checkout Session** | A Yuno server-side object that represents a payment attempt; contains amount, currency, customer, and configuration |
| **One-Time Token (OTT)** | A single-use token generated by the Yuno SDK after the customer fills in payment details; exchanged server-side for a payment |
| **Payment** | A Yuno object representing a charge attempt against a checkout session |
| **Payment Status** | The state of a payment: `SUCCEEDED`, `VERIFIED`, `APPROVED`, `PAYED` (success); `PENDING`, `PROCESSING`, `REQUIRES_ACTION` (pending); `REJECTED`, `DECLINED`, `CANCELLED`, `ERROR`, `EXPIRED`, `FAILED` (failure) |
| **3DS (3D Secure)** | Card authentication protocol; the SDK handles challenge flows and returns to the store via `callback_url` with `yuno_3ds_return` parameter |
| **APM (Alternative Payment Method)** | Non-card payment methods (wallets, bank transfers, etc.) rendered by the SDK in `#yuno-apm-form` |
| **Action Form** | Container (`#yuno-action-form`) for 3DS challenges and other interactive authentication flows |
| **Tokenization** | The process of replacing sensitive payment data with a one-time token via the Yuno SDK |
| **Idempotency Key** | Header (`x-idempotency-key`) sent to Yuno API to prevent duplicate payment creation |
| **Split Payment** | Marketplace feature where a payment is divided between a seller (PURCHASE) and platform (COMMISSION) |

## WooCommerce

| Term | Definition |
|---|---|
| **WC_Payment_Gateway** | WooCommerce base class for payment gateways; `Yuno_Gateway` extends this |
| **HPOS (High-Performance Order Storage)** | WooCommerce custom order tables feature (replaces post-based storage); plugin declares compatibility |
| **Block Checkout** | WooCommerce block-based checkout (default since WC 8.3); uses React components |
| **Legacy (Shortcode) Checkout** | Classic WooCommerce checkout using the `[woocommerce_checkout]` shortcode |
| **Order-Pay Page** | The `/order-pay/{id}/` page where the Yuno SDK is mounted; both checkout flows redirect here |
| **Receipt Page** | WooCommerce hook (`woocommerce_receipt_{gateway_id}`) used to render content on the order-pay page |
| **Order Key** | WooCommerce-generated key (`wc_order_xxx`) that authenticates access to a specific order's payment endpoints |
| **Order Meta** | Key-value storage on WC orders; the plugin uses `_yuno_checkout_session`, `_yuno_customer_id`, `_yuno_payment_id`, `_yuno_duplicate_order_id`, `_yuno_original_order_id` |
| **`payment_complete()`** | WooCommerce method that marks an order as paid, reduces stock, and triggers related hooks |
| **Order Status** | WC order lifecycle states: `pending` (awaiting payment), `processing` (paid, physical goods), `completed` (paid, digital/virtual), `failed`, `on-hold`, `refunded` |

## WordPress

| Term | Definition |
|---|---|
| **WP REST Nonce** | CSRF token (`wp_rest`) passed as `X-WP-Nonce` header; valid for both logged-in and guest sessions |
| **`wp_localize_script`** | WordPress function that injects PHP data as a JS global; used to pass `YUNO_WC` config to the frontend |
| **Transient** | WordPress temporary key-value store with TTL; used for concurrency locks (`yuno_webhook_lock_*`, `yuno_pay_lock_*`) |
| **`wp_remote_request`** | WordPress HTTP API function used by `yuno_wp_remote_json()` for all Yuno API calls |
| **Text Domain** | `yuno-payment-gateway`; used for i18n string wrapping with `__()` |

## Plugin-Specific

| Term | Definition |
|---|---|
| **Preflight Checks** | `runPreflightChecks()` in checkout.js -- runs before SDK init to redirect already-paid orders or auto-duplicate failed ones |
| **In-Place Retry** | After payment failure, the SDK is reset (`resetSdkState()`) and restarted on the same page without a redirect |
| **Auto-Duplicate** | Automatic creation of a new WooCommerce order when a previous payment failed, allowing seamless retry |
| **Backend Disagreement Remount** | When the SDK reports success but server-side verification (`/confirm`) disagrees, the SDK is reset and restarted |
| **Per-Order Customer** | Strategy where each WC order creates a fresh Yuno customer (`woo_order_{order_id}`) rather than reusing across orders |
| **Webhook Lock** | Transient-based lock (`yuno_webhook_lock_{order_id}`, 30s TTL) preventing race conditions between `/confirm` and webhook delivery |
| **`YUNO_WC`** | Global JavaScript object injected by `wp_localize_script` containing `restBase`, `nonce`, `orderId`, `orderKey`, `currency`, `total`, `country`, `email`, `language`, `publicApiKey`, `debug` |
